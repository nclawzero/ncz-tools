## A. Call-site switching to `trait AgentClient`

**The situation today:** `trait AgentClient` exists and is implemented by both `ZeroclawClient` and `OpenClawClient`, but every actual call site in `src/cli/commands.rs`, `src/cli/tui/repl.rs`, and `src/cli/tui/mod.rs` still holds `Arc<ZeroclawClient>` concretely. The trait isn't in the runtime path yet.

**What needs to change:**

1. `CommandHandler.client`: `Arc<ZeroclawClient>` → `Arc<tokio::sync::Mutex<Box<dyn AgentClient>>>` or similar.

   `submit_turn` on the trait takes `&mut self`. Shared access means either:
   - `Arc<Mutex<dyn AgentClient>>` — simplest, pay one lock per method call (negligible for a REPL)
   - `Arc<RwLock<dyn AgentClient>>` — slightly less contention but same lock overhead per call since most methods are serialized anyway
   - Interior mutability on each concrete type — spreads the complexity, not worth it

   Recommendation: `Arc<tokio::sync::Mutex<Box<dyn AgentClient + Send + Sync>>>`. Trait needs `+ Send + Sync` super-bounds.

2. Call sites touch `.lock().await` before every RPC. Most are one-line changes.

3. `ZeroclawClient`-specific inherent methods (`search_memory`, `memory_stats`, `list_cron_jobs`, cron helpers, `pairing`) stay as inherent methods on the concrete type. The `CommandHandler` either:
   - **Option 1**: carries a separate `mnemos_client` + `cron_client` reference that's specific to the concrete backend when available (cleanest — MNEMOS is orthogonal to agent anyway, per the scope memory)
   - **Option 2**: downcasts to concrete type for those calls (brittle)

   Option 1 is the right move. Aligned with the backend-scope guidance that MNEMOS is user-global.

4. REPL loop (`tui/repl.rs`): the `submit_turn` call becomes `client.lock().await.submit_turn(...)`. Lock held for the entire collect loop on openclaw; fine for single-active-turn UX.

**Testing:**
- Existing tests all pass unchanged if `CommandHandler::new` now accepts `Arc<Mutex<dyn AgentClient>>` — just wrap `ZeroclawClient` at construction.
- Add a smoke test with `OpenClawClient` behind the same `CommandHandler` to prove substitutability end-to-end, not just at the compile-time assertion.

**Risk:** Lock-held-across-await on `submit_turn` means no concurrent client ops during a chat turn. That matches today's UX (REPL blocks during a turn), so not a regression. Workspace support (chunk D) changes this — per-workspace lock, concurrent turns across workspaces.

## B. `session.tool` + tool-call rendering

**Background:** openclaw broadcasts two event types relevant to chat UX:
- `session.message` — transcript writes. Already consumed by `rpc_sessions_send_and_collect` (slice 3f).
- `session.tool` — tool-call activity. **Currently dropped** by the event router because the read loop delivers it on `event_rx` but no consumer reads it.

`AssistantContentPart::ToolUse` and `::ToolResult` are parsed but only surface via `is_tool_only()` for the "keep waiting" heuristic (slice 3i). They aren't shown to the user.

**What the REPL should show:**

A typical agent turn:
```
[thinking]       ← optional, folded by default
[tool: shell]    ← tool_use part
  command: echo hi
[tool result]    ← from the tool_result part or session.tool event
  hi
[assistant]      ← final text
Hi there — I ran `echo hi` for you.
```

**Implementation sketch:**

1. Change `rpc_sessions_send_and_collect` to return a richer structure than `String`:
   ```rust
   pub struct TurnResult {
       pub text: String,           // display_text of final assistant message
       pub tool_calls: Vec<ToolCall>,
       pub thinking: String,       // if the user wants to see it
       pub run_id: String,
   }
   ```
2. Keep the current `String`-returning shape as a thin wrapper for backward compat with the trait: `submit_turn` uses `TurnResult::text`.
3. Collection loop accumulates tool-call parts across messages (not just the final text message) and returns them together. Needs ordered accumulation: text → tool_use → (tool_result) → text.
4. Optionally subscribe to `session.tool` events for real-time tool-running indication. Less critical — `tool_use` parts in `session.message` already carry the arguments; server-side execution progress is nice-to-have, not required for UX.
5. REPL rendering: a small formatter that takes `TurnResult` and emits the boxed layout above. ANSI color for the `[tool: name]` headers.

**Scope note:** the trait stays `submit_turn -> String` for simplicity; tool surfacing is an OpenClawClient-specific enrichment exposed via `rpc_sessions_send_and_collect_rich` (or similar name) that the openclaw-workspace renderer calls directly.

## C. Model/provider population on loaded sessions

`AgentClient::load_session` on openclaw currently fills `model: String::new(), provider: String::new()` because the `GatewaySessionRow` doesn't carry that metadata.

Openclaw derives a session's model from:
1. The session entry's `model` override (if set at creation)
2. Fallback to the agent's default model (`agents.<id>.defaults.model.primary`)

Neither is exposed by `sessions.list`. Two paths:

**Path C1 — Single config fetch:** Call `config.get` (if exposed) or equivalent once at connect, cache the agent-id → model mapping. When `load_session` parses a key like `agent:main:<session>`, look up the `main` agent's default model from the cache. Good for sessions that use the agent default; misses per-session overrides.

**Path C2 — Per-session probe:** Add a new helper RPC, probably something like `agents.show(id)` or reading the session entry's `preferredModel` field (which IS in the entry but isn't in the list row). Requires trying an RPC we haven't used yet; may or may not exist. Worth 30 minutes of openclaw source-digging before committing.

Neither path is critical for current REPL UX (status bar just shows empty strings) — punt until it causes a visible bug.

## D. Multi-workspace runtime switching

**The actual v0.2 payoff.** The whole reason the trait exists.

### UX surface

- Tab bar across the top of the TUI showing active workspaces:
  ```
  [1:zeroclaw@typhon]*  [2:openclaw@typhon]  [3:zeroclaw@edge]
  ```
  Active workspace highlighted. Cycle + jump hotkeys:
  | Binding | Action |
  |---|---|
  | `Ctrl+T` | New workspace (prompts for backend + URL, or picks from config) |
  | `Ctrl+Tab` / `Ctrl+Shift+Tab` | Cycle forward / back |
  | `Alt+1` … `Alt+9` | Jump to workspace N |
  | `Ctrl+W` | Close current workspace (prompt if busy) |
  | Plain `Tab` | **Stays as slash-command completion** — don't repurpose |
  | `/workspace new\|switch\|close\|list` | Slash-command mirror of hotkeys for scripting + discoverability |

- Per-workspace state: its own client, session, scrollback, model.
- Shared state: MNEMOS client is single-instance across all workspaces (MNEMOS is user-global per the scope memory). All `/memory *` commands operate on the same store regardless of active workspace.

### Data model

```rust
struct Workspace {
    id: usize,
    label: String,
    client: Arc<Mutex<Box<dyn AgentClient + Send + Sync>>>,
    session: Session,
    scrollback: Vec<Turn>,
    stream_state: Option<RunningTurn>,
}

struct App {
    workspaces: Vec<Workspace>,
    active: usize,
    shared_mnemos: Option<MnemosClient>,
    config_path: PathBuf,
}
```

### Config file

`~/.zterm/config.toml`:

```toml
active = "zeroclaw-typhon"

[[workspaces]]
name = "zeroclaw-typhon"
backend = "zeroclaw"
url = "http://127.0.0.1:42617"
token_env = "ZEROCLAW_TOKEN_TYPHON"

[[workspaces]]
name = "openclaw-typhon"
backend = "openclaw"
url = "ws://127.0.0.1:18789"
# openclaw uses device key; no token_env needed unless gateway.auth != "none"

[[workspaces]]
name = "zeroclaw-edge"
backend = "zeroclaw"
url = "http://192.168.207.62:42617"
token_env = "ZEROCLAW_TOKEN_EDGE"
```

### Background streams

When a workspace is active, its events render inline. When backgrounded, events still arrive (read-loop tasks keep running) and accumulate in that workspace's scrollback. Switching back reveals accumulated output. Badge on tab bar when a backgrounded workspace has new content: `[2:openclaw@typhon●]`.

### Failure modes

- Workspace client loses connection: tab badge turns red, workspace becomes read-only, user can hit `Ctrl+R` to reconnect or `Ctrl+W` to close. Other workspaces unaffected.
- Workspace client is mid-turn when user switches away: streaming continues into the backgrounded scrollback. Returning to the workspace shows the accumulated content.
- Ctrl+Tab eaten by some terminal emulators (Terminal.app, default iTerm profile): `Alt+N` is the portable fallback. Ship both.

### Scope breakdown (order of implementation)

1. **D1 — App model + config** (½ session): Pure data. `App`, `Workspace`, config parsing, per-workspace instantiation. No UX yet.
2. **D2 — Tab bar rendering** (½ session): ratatui widget above the chat pane.
3. **D3 — Hotkeys + slash commands** (½ session): key routing, workspace switching, new/close.
4. **D4 — Background streaming** (1 session): spawn per-workspace event loop, accumulate into scrollback even when backgrounded, badge dirty tabs.
5. **D5 — Error recovery + reconnect** (½ session): workspace-level retry, clean teardown on close.

All of this depends on chunk A (call-site switching). D1 can start after A lands.

## Recommended sequence

1. **A** — unblocks everything else, smallest scope, biggest downstream leverage.
2. **B** — quick UX win; no dependencies.
3. **D** — start the 5-step chapter; roughly 2–3 focused sessions total.
4. **C** — fold in as polish when C1 becomes trivially useful.

Suggested target: v0.2.0 cut after A + B + D1-D3 land. v0.2.1 adds D4-D5. v0.3 adds a third backend (nemoclaw) and the model/provider population story.

## Not in v0.2 (deliberate punt)

- OpenAI-compatible backends (Hermes etc.) — excluded per `project_zterm_backend_scope` memory.
- In-terminal attachment rendering (file uploads, image preview) — v0.3 at earliest.
- Multi-user collaboration in a single workspace — explicit non-goal; zterm is single-user.
- Any kind of plugin/extension system — doctrine: "light, thin, fast, memory-safe". Plugins are a future fork decision.
